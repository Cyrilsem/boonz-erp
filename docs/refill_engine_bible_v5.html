Data layer — the four oracles

Four different questions, four different sources. Never confuse them.

Each oracle answers exactly one question about the fleet. The refill engine must query the right one for the right question — mixing them is the #1 source of bad decisions.

📦 Stock oracle — "What's physically on the shelf right now?"
Source: v_live_shelf_stock view, flattening weimi_device_status.door_statuses JSONB
Status: ✅ Live. CC-00 complete. View created Round 1, hotfixed for cabinet selector bug. Verified against NOOK ground truth.

How it works: The refresh-stage1 Edge Function calls Weimi's /ext/device-info endpoint and stores the response as JSONB in weimi_device_status.door_statuses. Structure is array-of-cabinets, each cabinet has layers, each layer has aisles. The view does DISTINCT ON (machine_id) ORDER BY snapshot_at DESC to get only the latest snapshot per machine, then flattens via jsonb_array_elements down to one row per slot.

Columns exposed: machine_id, machine_name, weimi_device_id, cabinet_index, layer_label, aisle_code, slot_name, goods_name_raw, pod_product_id (4-tier match), match_method, current_stock (clamped to ≥0), max_stock, fill_pct, is_broken, is_enabled, price_aed, snapshot_at, is_eligible_machine.

Match resolution: 4-tier pipeline. Direct exact match → case-insensitive match → product_name_conventions alias lookup → unmatched (NULL pod_product_id). Current stats: 87% direct, 6% conventions, 1% case-insensitive, 6% unmatched (only 1 real test product on non-eligible machines).

Refresh cadence: nightly at 23:59 Dubai via cron, plus on-demand via SELECT refresh_fleet_data(90).
⏳ Expiry oracle — "What's about to go bad in the pods?"
Source: pod_inventory, driver-maintained + FIFO sales decrement
Status: ✅ Live. Pre-existing, not modified in Round 1.

How it works: Driver does monthly audit, writes batch + expiration_date rows. refresh-stage1 calls run_pod_inventory_decrement() which subtracts new sales from the oldest-expiring matching batch. So current_stock in this table reflects "what's left in the pod by batch and expiry," not "what's on the shelf."

Consumers: Engine 1 (catalog-level expiry-at-risk list), Engine B (expiry-aware quantity sizing), Engine 2 (rescue move prioritization for batches near expiry).

NOT used for: "how much is in the machine right now" — that's the stock oracle. Confusing these two was the original bug we hit on day 1.
💰 Sales oracle — "What sold, when, and for how much?"
Source: sales_history, live from Weimi via refresh-stage1
Status: ✅ Live. 16,661 rows as of 2026-04-09. Latest transaction within 1 minute of every refresh.

How it works: Weimi's /ext/history-order-page (90 days default lookback) and /ext/today-order-page endpoints. Pulled, deduplicated by tradeNoIn, upserted via upsert_sales_lines RPC.

Consumers: Lifecycle scoring (evaluate-lifecycle nightly), velocity calculation, demand forecasting, correlation matrix.
🎯 Outcome oracle — "Did the refill plan match reality?"
Source: refill_dispatching, populated by field team
Status: ✅ Live. Pre-existing. NEW in v5: explicitly recognized as the learning loop's outcome source.

How it works: When the operator approves a refill plan on /refill, lines flow into refill_dispatching. Driver picks them up via /field/Daily Refill, marks each as packed/picked_up/filled_quantity/returned/return_reason. Engine D's learning loop reads these outcomes via the decision_log.outcome_dispatch_id FK to know whether its quantity sizing, swap candidates, and clubbing decisions actually worked on-site.

Consumers: Engine B (learn from filled_quantity vs sent_quantity gap), Engine C (learn from returned items + return_reason), Engine D (overall learning loop).

Open issues found during implementation

Things we discovered while building Round 1 but haven't fully fixed yet. Each one tracked here so we don't lose them.

A8 over-stocked at NOOK (9/8 = 112%)
After today's refill, NOOK A8 was swapped from "Caprice" to "SF Pancake" but the new product's maxStock wasn't updated to match the actual physical capacity. Result: current_stock exceeds max_stock, fill_pct > 100%. Not a view bug — this is real data from Weimi.

Implication: Engine B (quantity sizing) must handle fill_pct > 100 as a valid state meaning "max_stock is out of date, flag for operator review, don't try to refill." The view returns the over-stocked state honestly; downstream consumers must not crash or wrap around.
Product swap detection at slot level
NOOK A8 went from "Caprice" → "SF Pancake" between snapshots. slot_lifecycle is keyed on (machine_id, shelf_code), not product, so the Caprice lifecycle history silently carries over to SF Pancake. This is contamination analogous to ALHQ → JET-1016 but at the slot level.

Fix needed: evaluate-lifecycle (or a new pre-step) should detect goods_name_raw changes between snapshots and reset the slot's lifecycle score for the new product (with grace period). Add to Phase 1 backlog.
wait_for_request() times out for long-running Edge Functions
refresh-stage1 takes 60-90 seconds. pg_net's default request timeout is shorter (~10s), so net._http_response.status_code stays NULL even though the Edge Function actually completed successfully (visible in get_logs). The function still works (data lands in DB), but our wait_for_request() helper can't programmatically detect completion.

Workaround in use: poll the actual data freshness (e.g., MAX(weimi_device_status.snapshot_at)) instead of net._http_response.

Proper fix later: rewrite wait_for_request() to take a "freshness signal" parameter — a SQL fragment that returns a timestamp — and poll that instead. Low priority. Current workaround is fine.
JET-1016 placeholder fields after Round 2 split
Round 2 CC-09 will split ALHQ → JET-1016 using the repurpose_machine() function. CC doesn't know the real values for JET-1016's location_name, location_type, source_of_supply, so it uses placeholders ("JET location (operator to confirm)", "coworking", "BOONZ").

Action after Round 2 lands: CS to update these fields with real values via the admin UI.
5 other already-repurposed machines never investigated
Earlier query found 5 machines with repurposed_at IS NOT NULL in the DB (besides the ALHQ → JET-1016 case being handled in Round 2). They're all Switched off + repurposed_at set, which is a healthy state, but we never checked whether any of them have lingering data contamination on their old machine_ids. Audit needed after Round 2.
"Product for testing only" still in v_live_shelf_stock
35 slot rows on ACTIVATEMCC-1024 and WH1-2002 (both non-eligible machines). Product is intentional test placeholder, not a real SKU. Currently quarantined by is_eligible_machine = false filter, so it can't leak into refill decisions.

Action: none needed for refill engine. CS may want to clean up the test placeholder in Weimi at some point but it's not blocking anything.

Lessons learned

Engineering lessons from Phase 0. Each one is a personal-rulebook addition for future prompts.

JWT-in-migration regression (commits 747612c → 220af7e → 0da2861 → cea5aee)
CC, when given an ambiguous prompt about Edge Function auth, "helpfully" hardcoded the anon key into the SQL function body and granted EXECUTE to authenticated. Took 3 follow-up commits to clean up: revert function body to vault read, REVOKE authenticated grants explicitly (because CREATE OR REPLACE FUNCTION preserves ACLs), drop the Authorization: Bearer header (which refresh-stage1 validates as a user token, not a service role token).

Personal rulebook addition: for any prompt that involves credentials, secrets, or auth design, include an explicit "DO NOT alter the following design choices under any circumstances, even if it would simplify the implementation" section listing the load-bearing design decisions.
CREATE OR REPLACE FUNCTION does not reset ACLs
The function body changes, but pre-existing GRANT/REVOKE rules persist. So if a previous migration granted EXECUTE to a role and a new migration replaces the function body without explicitly revoking that grant, the role still has access.

Personal rulebook addition: any migration that uses CREATE OR REPLACE FUNCTION on a security-sensitive function MUST follow it with explicit REVOKE statements for every role that should not have access, plus the GRANTs for the roles that should.
JSONB structure assumptions (door_statuses cabinet selector bug)
I told CC the door_statuses structure was "cabinets → layers → aisles" in shorthand. CC interpreted that as door_statuses.cabinets[] when actually door_statuses IS the cabinets array (no wrapper key). Result: empty view on first build.

Personal rulebook addition: when describing JSONB structure in prompts, always specify the literal top-level type explicitly. Use phrases like "door_statuses is an array of cabinet objects, length N, where each element is...". Never use the conceptual shorthand "X → Y → Z" without specifying the JSON literal at each level.
Verify capabilities before promising to handle a task
I told you "I'll fetch the service role key via my Supabase MCP tools" and then discovered my MCP tools only expose publishable keys (anon + sb_publishable_*), not service_role. By design — which is good security. But I made the promise without checking the tool catalog first, costing us a round-trip.

Personal rulebook addition: before offering to do a step that requires a specific tool capability, verify the tool can actually do it. Especially for security-sensitive operations where the answer might be "no, by design."
Match the existing codebase's calling convention
refresh-stage1 is called by the n8n operational-signals workflow with apikey-only headers, no Authorization. When I wrote the SQL wrapper, I added a "belt-and-suspenders" Authorization: Bearer header that turned out to be the cause of the 401 (because refresh-stage1's auth code path validates Authorization as a user JWT). The fix was to remove the header and match n8n's pattern.

Personal rulebook addition: when writing a new caller for an existing function, replicate the exact calling convention used by other working callers. Don't add headers, parameters, or auth fields that the existing callers don't send. The function's auth logic was designed around the existing caller pattern.

Operational runbook

Common operations and how to perform them. Add new entries as we discover them.

How to refresh fleet data on demand
From Claude chat (or any service_role SQL session): 
SELECT public.refresh_fleet_data(90) AS request_id;
-- Returns a request_id (bigint)

-- Wait ~90 seconds, then verify by checking data freshness:
SELECT MAX(snapshot_at) FROM weimi_device_status;
-- Should be within last 60 seconds

-- Or check the Edge Function logs directly:
-- (via Supabase MCP get_logs tool, service='edge-function')
Do NOT use wait_for_request() for refresh-stage1 — pg_net's request timeout truncates the response tracking even though the function completes. Poll weimi_device_status.snapshot_at instead.
How to rotate the vault secret
If the Supabase service_role key gets rotated (or compromised): 
-- 1. Rotate in Dashboard → Project Settings → API Keys
-- 2. Update the vault entry with the new key:
SELECT vault.update_secret(
  (SELECT id FROM vault.secrets WHERE name = 'edge_function_service_key'),
  '<new service_role JWT>'
);
-- 3. Verify:
SELECT * FROM public.check_edge_function_service_key();
-- Should still say 'present'
-- 4. Smoke test:
SELECT public.refresh_fleet_data(90);
No code changes needed. The wrapper functions read from the vault on every call.
How to add a new product alias
When the operator confirms that a product appearing in v_live_shelf_stock with match_method='unmatched' is actually an existing pod_product under a different name: 
INSERT INTO public.product_name_conventions (original_name, official_name, mapped_at)
VALUES ('<Weimi name>', '<pod_products.pod_product_name>', CURRENT_DATE);

-- Verify the alias resolves on next query of v_live_shelf_stock
Through Claude Code as a committed migration, not ad-hoc, to keep repo and DB in sync.
How to add a new venue group member

How to repurpose a machine cleanly (post-CC-08)
After Round 2 lands, never rename a machine in place. Instead: 
-- Via SQL (or via the admin UI button when CC-08 lands):
SELECT * FROM public.repurpose_machine(
  p_old_machine_id := '<current machine_id>',
  p_new_official_name := 'NEW-NAME-XXXX-Y0',
  p_new_location_name := 'New venue name',
  p_new_pod_location := 'NEW-NAME-XXXX-Y0',
  p_new_location_type := 'cinema',  -- or coworking, office, retail, etc.
  p_new_building_id := 'optional building tag',
  p_new_source_of_supply := 'BOONZ',  -- or VOX, LLFP
  p_new_venue_group := 'INDEPENDENT'  -- or ADDMIND, VOX, etc.
);
-- Returns: old_machine_id, new_machine_id, slots_archived, result
This atomically marks the old row as repurposed (repurposed_at set, adyen Switched off, weimi_device_id detached), archives old slot_lifecycle rows, and creates a fresh machine row with the new identity inheriting the same physical Weimi device.

Design principles

  1. One direction, top to bottom. No loops within a run. Engine 1 → 2 → Layer 1 → Step 2 → Stage 3 (A, C, B) → Engine D → Engine E.
  2. Four oracles, four questions. Stock = v_live_shelf_stock. Expiry = pod_inventory. Sales = sales_history. Outcomes = refill_dispatching. Never confuse them.
  3. Engine 1 is portfolio-only. Catalog decisions, not slot decisions.
  4. Layer 1 stays pure. Per-machine priority only. Geography enters at Step 2.
  5. Guardrails apply at every engine. Each engine reads only the types relevant to its decision.
  6. Engine D is the only writer. No other engine commits to refill_plan_output.
  7. Shared fit scorer. Engine C and Engine 2 share the same scoring function for consistency.
  8. Stage 0 calculators are reference data, not decisions.
  9. Learning loop is across runs only. Deterministic, debuggable.
  10. Reason codes are mandatory. Every override tagged. Engine D's learning ignores operational reasons (stock_unavailable, logistics_constraint).
  11. Advisor-only forever. Skill never bypasses /refill review.
  12. Downstream pipeline unchanged. Skill writes to refill_plan_output, that's it.
  13. Machine eligibility is a precondition. Decommissioned, repurposed, and non-live machines are filtered out before Layer 1 even runs.
  14. Vault for all secrets. Never embed JWTs in any migration or function body.
  15. Bible updates at every checkpoint. Bible drift is the failure mode we cannot afford.
