Boonz ERP · Architecture · Foundational Document

Backend Constitution

The invariants the Boonz backend must hold. Every change is measured against these.
Version 1.0 Effective 2026-04-25 Owner CS Status Draft for ratification

Why this exists. The Boonz ERP has accumulated multiple write paths to the same entities, dormant tables, deprecated RPCs that still execute, and parallel admin surfaces between /app and /field. The cause is not bad code; it is the absence of a written contract. This document is that contract. It makes the wrong thing impossible by writing the rules into the database and the CI pipeline. From the moment it ratifies, every PR either enforces an article, violates one (rejected), or proposes a new one (deliberate). Patches stop being cheap.

I · Write Paths

Article 1

One canonical write path per entity

For every protected entity (Appendix A), there is exactly one SECURITY DEFINER RPC that performs writes. The UI, edge functions, n8n, and cron all call only that RPC.

Why
Multiple write paths are how silent drift happens. We currently have three UIs writing machines directly while a canonical add_new_machine() RPC sits unused.
Enforcement
CI lint blocks .update("<protected>") and .insert("<protected>") outside of the RPC layer. REVOKE INSERT, UPDATE, DELETE ON <protected> FROM authenticated.
Article 2

RLS is mandatory on protected tables

Every table holding user data, financial data, audit logs, or operational state has Row Level Security enabled with explicit policies for each role that may read or write.

Why
"Append-only by convention" is not append-only. pod_inventory_audit_log currently has 2,887 rows and no RLS — any authenticated user can DELETE the audit trail.
Enforcement
pg_class.relrowsecurity = true for every table in Appendix A. Supabase advisor rls_disabled_in_public check must pass weekly.
Article 3

The `authenticated` role is write-restricted

The authenticated role has SELECT-only on protected tables (subject to RLS). All INSERT/UPDATE/DELETE goes through SECURITY DEFINER RPCs.

Why
When the role can write directly, RLS policies must cover every column-level guard themselves; this is brittle and incomplete in practice.
Enforcement
REVOKE INSERT, UPDATE, DELETE ON <protected> FROM authenticated is part of the migration that creates each canonical RPC.

II · Validation

Article 4

SECURITY DEFINER functions explicitly validate

Every SECURITY DEFINER function checks the caller's role/claim, validates inputs against business rules, and sets app.via_rpc = 'true' and app.rpc_name = '<name>' for the duration of its work.

Why
DEFINER without validation is just a privilege escalation. The session vars are how the write-audit trigger distinguishes RPC writes from direct writes.
Enforcement
Function lint: every SECURITY DEFINER body must contain set_config('app.via_rpc', 'true', true) and at least one auth.uid() or auth.jwt() reference.
Article 5

Status fields are state machines

Status columns (machines.status, slots.status, pod_inventory.status, warehouse_inventory.status, etc.) only accept transitions defined in the RPC layer. The CHECK constraint enumerates valid values; the RPC enumerates valid transitions.

Why
Free-form status writes are how machines end up in invalid states ("Pending" with no warehouse, "Active" with no Adyen store).
Enforcement
No FE writes status directly. Each protected entity has a set_<entity>_status(target_state) RPC that validates the transition from the current state.
Article 6

Manager-only fields are manager-only everywhere

warehouse_inventory.status is set only by the warehouse manager via the canonical RPC. No trigger, function, cron, n8n sync, or app flow may write it.

Why
Status drives refill-engine eligibility. Auto-mutation silently corrupts the refill pool. (Existing rule, formalized.)
Enforcement
Trigger raises EXCEPTION if the column is changed by any source other than the canonical RPC, identified via app.rpc_name = 'set_warehouse_status'.

III · Audit

Article 7

Audit logs are append-only

Tables ending in _audit_log and tables tagged append_only in Appendix A have RLS policies allowing INSERT only. UPDATE and DELETE are blocked for every role, including service_role (which uses a separate retention RPC for legitimate purges).

Why
An audit log that can be rewritten is not an audit log.
Enforcement
CREATE POLICY no_update FOR UPDATE USING (false) and the equivalent for DELETE. Retention is performed by a dedicated SECURITY DEFINER RPC that records its own deletion event before deleting.
Article 8

Every RPC writes to write_audit_log

Every canonical write RPC inserts a row into write_audit_log containing caller UID, caller role, action, target table, target PK, before/after JSON, and timestamp before returning.

Why
This is the data feed for the future Data Health tab. It also makes incident reconstruction possible.
Enforcement
Helper audit_log_write(p_op, p_table, p_pk, p_before, p_after) is called from every RPC. Lint flags RPCs that don't.

IV · Surfaces

Article 9

Edge functions are stateless

Edge functions authenticate the caller, validate input shape, and call an RPC. No business logic, no direct table writes, no caching of business state.

Why
Logic in edge functions is invisible to the database — it cannot be enforced by RLS, cannot be replayed, cannot be audited consistently.
Enforcement
Edge function lint: no .from(...)/.update/.insert/.delete calls outside of .rpc(...).
Article 10

n8n calls the RPC layer

All n8n write workflows go through SECURITY DEFINER RPCs using a dedicated n8n_writer service token. n8n does not perform direct table writes.

Why
When n8n bypasses the RPC layer, the audit log misses entire categories of writes (Weimi sales, stock refresh, etc.).
Enforcement
The n8n_writer role has EXECUTE on canonical RPCs but no INSERT/UPDATE/DELETE on protected tables.
Article 11

Cron jobs call RPCs

pg_cron jobs invoke SECURITY DEFINER RPCs only. No UPDATE/INSERT SQL appears in a cron.job schedule directly.

Why
Cron jobs that mutate tables directly are a hidden surface that nobody reviews. They have been the source of silent failures (e.g. evaluate-lifecycle stale since 2026-04-10).
Enforcement
Audit query checks cron.job.command for non-RPC mutation patterns; flagged jobs are rewritten.

V · Schema Hygiene

Article 12

Migrations are forward-only and idempotent

Every migration is forward-only. To "rollback" a change, write a new forward migration that reverses it. Migrations use IF NOT EXISTS / IF EXISTS so re-running them is safe.

Why
Reverting migrations is how branches diverge from prod. Forward-only with idempotency means any environment can be replayed from migration history.
Enforcement
PR template requires "what does this change" and "how to undo" (which is the next forward migration). No git revert on merged migration commits without a follow-up forward migration.
Article 13

Deprecation has a process

When an RPC, table, or column is deprecated: (1) marked deprecated in the migration, (2) SECURITY DEFINER changed to SECURITY INVOKER for functions, (3) EXECUTE revoked from authenticated, (4) pg_stat_user_functions monitored for 90 days. Only after zero calls for 90 days is the object dropped.

Why
Surprise removals break n8n and external integrations we did not know about.
Enforcement
A "Deprecated registry" lives in docs/architecture/deprecated.md with each item, deprecation date, expected removal date, and last-call timestamp.
Article 14

No snapshot tables when a view suffices

New tables that materialize a query result for performance reasons require ADR signoff. Existing snapshot tables (machine_summary, refill_dispatch_plan, daily_pipeline_runs, etc.) are scheduled for retirement in favor of views or materialized views with explicit refresh semantics.

Why
Snapshot tables go stale silently. machine_summary is currently 44 days stale, refill_dispatch_plan dead since 2026-04-01, with FE pages still pointing at them.
Enforcement
Snapshot-table audit weekly; any table whose latest row is >7 days stale is flagged in the Data Health tab.

VI · Process

Article 15

Every PR declares invariants and CI enforces

Every PR description names which articles it touches (or "none"). CI runs three governance checks on every PR: protected-write-lint, rls-coverage, rpc-audit-coverage.

Why
Governance without enforcement is decoration. Lint catches drift before it ships, not in retrospect at audit time.
Enforcement
.github/workflows/governance.yml runs on every PR. Failures block merge. Bypass requires a one-line ADR pinned in the PR.

Appendix B — CI Checks (v1)

Three checks run on every PR via .github/workflows/governance.yml:

  • protected-write-lint — fails if the diff contains supabase.from("<protected>").update|insert|delete outside of supabase/functions/ or paths whitelisted in docs/architecture/lint-allowlist.md.
  • rls-coverage — fails if any table in Appendix A has relrowsecurity = false after migrations are applied to the PR's preview branch.
  • rpc-audit-coverage — fails if any new SECURITY DEFINER function does not call audit_log_write() at least once.

Appendix C — Amendment Process

This Constitution is amended via PR. Each amendment must include:

  • The article being added, removed, or modified.
  • A one-paragraph rationale (why the existing article is insufficient).
  • The migration or CI change that enforces the new article.
  • Signoff by the codebase owner (CS) before merge.

Amendments take effect at merge. Existing code is brought into compliance via the next phase plan, not retroactively in the same PR.
