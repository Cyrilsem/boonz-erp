What this means in plain language

Before: the planogram table accepted reads and writes from anyone with a valid login because RLS was switched off. There was no enforcement layer the database could check.

After: RLS is on, with four policies that currently allow every authenticated user to read, insert, update and delete — the same effective permissions as before. Nothing in the FE changes. What changed is structural: the framework is now active, so in Phase B we can tighten one policy at a time (e.g., "UPDATE only allowed when called via the canonical RPC") without ever touching the rows themselves.

Think of it as installing the door frame. The door is still wide open. But now there's somewhere to hang the lock.

What this means in plain language

Before: the audit log was append-only in name only. Anyone with database access could DELETE rows from the audit trail — which defeats the purpose of having an audit log in the first place.

After: the audit log is now genuinely append-only. INSERT still works (the existing trigger that writes to it continues to function). UPDATE and DELETE are blocked at the database layer for every role. The only writer in the system today (auto_decrement_pod_inventory) runs with elevated privileges so it bypasses RLS — which is correct, because it's the legitimate writer. Everything else is now locked out.

Think of it as turning a paper logbook into one of those bound notebooks where you can't tear pages out without leaving evidence. New entries still go on. Old entries can't disappear.

What was actually executed

Migration applied · phaseA_a1_rls_planogram_pia

-- planogram: enable RLS + 4 permissive policies (no behavior change)
ALTER TABLE public.planogram ENABLE ROW LEVEL SECURITY;
CREATE POLICY planogram_select ON public.planogram FOR SELECT TO authenticated USING (true);
CREATE POLICY planogram_insert ON public.planogram FOR INSERT TO authenticated WITH CHECK (true);
CREATE POLICY planogram_update ON public.planogram FOR UPDATE TO authenticated USING (true) WITH CHECK (true);
CREATE POLICY planogram_delete ON public.planogram FOR DELETE TO authenticated USING (true);

-- pod_inventory_audit_log: enable RLS + append-only policies (UPDATE/DELETE blocked)
ALTER TABLE public.pod_inventory_audit_log ENABLE ROW LEVEL SECURITY;
CREATE POLICY pial_select ON public.pod_inventory_audit_log FOR SELECT TO authenticated USING (true);
CREATE POLICY pial_insert ON public.pod_inventory_audit_log FOR INSERT TO authenticated WITH CHECK (true);
CREATE POLICY pial_no_update ON public.pod_inventory_audit_log FOR UPDATE USING (false);
CREATE POLICY pial_no_delete ON public.pod_inventory_audit_log FOR DELETE USING (false);
