# PRD-011: Fix FE Refill Commit Flow

**Status:** Ready for implementation
**Priority:** P0 (blocking production commits)
**Date:** 2026-05-25
**Author:** CS
**File:** `src/app/(app)/refill/RefillPlanningTab.tsx`

---

## Context

The FE refill planning page (`/refill`) lets CS load a pod-level draft, edit quantities inline, remove rows, and commit (approve + finalize + stitch + dispatch) in one click. The commit flow broke in production on 2026-05-25 with "Stitch failed: no approved rows." Root cause investigation revealed 5 bugs in `RefillPlanningTab.tsx`, all in the commit chain. CS had to fall back to a manual chat-based Path B to commit today's plan.

---

## Bug 1: Date mismatch in commit chain

**Severity:** P0 (blocks every commit when draft date != selectedDate)
**Location:** `commitDraft` callback, lines 372-438

**Problem:**
`commitDraft` passes `selectedDate` (a prop from the parent, defaulting to `CURRENT_DATE + 1`) to all four RPC calls: `approve_pod_refill_plan`, `engine_finalize_pod`, `stitch_pod_to_boonz`, and `approve_refill_plan`.

The draft is generated by the 8pm cron for a specific `plan_date`. When CS opens the page the next morning, `selectedDate` may not match the draft's actual `plan_date`. For example: the cron ran at 8pm on May 24 and generated a draft for May 25. CS opens the page on May 25, but `selectedDate` defaults to May 26. The commit chain calls `approve_pod_refill_plan('2026-05-26')`, which finds 0 draft rows (they're for May 25), so stitch correctly fails with "no approved rows."

**Fix:**

1. When `loadDraft` / `get_pod_refill_draft` returns rows, extract and store the actual `plan_date` from the response in a new state variable: `draftPlanDate`.
2. `commitDraft` must use `draftPlanDate` (not `selectedDate`) for all RPC calls.
3. Add a guard: if `draftPlanDate` is null/empty, disable the Commit button and show "No draft loaded."
4. Optional: show a warning banner if `selectedDate !== draftPlanDate` so CS knows the loaded draft is for a different date than the date picker shows.

**Acceptance criteria:**

- AC1: Commit uses the loaded draft's plan_date, not the date picker value.
- AC2: Commit button is disabled when no draft is loaded.
- AC3: If selectedDate differs from draftPlanDate, a yellow warning banner appears: "Draft loaded for {draftPlanDate}, but date picker shows {selectedDate}."

## Bug 2: Inline edits not persisted before commit

**Severity:** P0 (silently discards all CS edits)
**Location:** `commitDraft` callback, lines 372-438; state declarations at lines 145-148

**Problem:**
When CS edits a quantity inline, the new value is stored in React state `editedQty: Record<number, number>` (line 145). When CS strikes through a row to remove it, the row index is added to `removed: Set<number>` (line 148). Neither of these client-side changes are persisted to the database before the commit chain runs.

`commitDraft` calls: approve -> finalize -> stitch -> dispatch. It never calls `edit_pod_refill_row` to write the qty changes or `stop_pod_refill_row` to mark removed rows. The stitch therefore uses the original engine-generated quantities, silently ignoring everything CS changed.

**Fix:**
Insert a "Step 0: Persist edits" block at the start of `commitDraft`, before Gate 1:

```
// Step 0a: Persist quantity edits
for (const [rowIndex, newQty] of Object.entries(editedQty)) {
  const row = planRows[Number(rowIndex)];
  if (!row) continue;
  await supabase.rpc('edit_pod_refill_row', {
    p_plan_date: draftPlanDate,       // from Bug 1 fix
    p_machine_id: row.machine_id,
    p_shelf_id: row.shelf_id,
    p_pod_product_id: row.pod_product_id,
    p_action: row.action,
    p_new_qty: newQty,
    p_reason: 'FE inline edit',
    p_conductor_session: null
  });
}

// Step 0b: Persist row removals (stop rows)
for (const rowIndex of removed) {
  const row = planRows[rowIndex];
  if (!row) continue;
  await supabase.rpc('edit_pod_refill_row', {
    p_plan_date: draftPlanDate,
    p_machine_id: row.machine_id,
    p_shelf_id: row.shelf_id,
    p_pod_product_id: row.pod_product_id,
    p_action: row.action,
    p_new_qty: 0,
    p_reason: 'FE row removal',
    p_conductor_session: null
  });
}
```

**Acceptance criteria:**

- AC1: Every `editedQty` entry is written to `pod_refill_plan` via `edit_pod_refill_row` before approve is called.
- AC2: Every `removed` row is set to qty=0 via `edit_pod_refill_row` before approve is called.
- AC3: If any edit RPC fails, the commit chain aborts with a clear error message naming the failed row.
- AC4: After successful persist, clear `editedQty` and `removed` state.

## Bug 3: No restore-from-superseded mechanism

**Severity:** P1 (UX gap, workaround exists via chat)
**Location:** FE row rendering (superseded rows show "Restore" button, but it only toggles client-side `removed` Set)

**Problem:**
When `engine_finalize_pod` runs conflict resolution, it can mark rows as `status='superseded'`. The FE displays these rows with a "Restore" button. However, clicking "Restore" only removes the row index from the `removed` Set (a client-side toggle). There is no RPC to flip a superseded row back to `status='draft'` in the database. If CS wants to restore a superseded row and commit, the restore never reaches the DB.

**Fix:**

1. Create a new RPC `restore_pod_refill_row(p_plan_date, p_machine_id, p_shelf_id, p_pod_product_id, p_action)` that flips `status='superseded'` back to `status='draft'`.
2. Wire the FE "Restore" button to call this RPC.
3. On success, update the local `planRows` state to reflect the new status.

**Acceptance criteria:**

- AC1: RPC `restore_pod_refill_row` exists and flips superseded -> draft.
- AC2: RPC rejects if the row is not in 'superseded' status.
- AC3: FE "Restore" button calls the RPC and updates local state on success.

## Bug 4: 8pm cron not firing

**Severity:** P1 (blocks automated draft generation)
**Location:** Supabase cron job #13 (`phaseF_stage1_prep_8pm_dubai`, schedule `0 16 * * *` UTC)

**Problem:**
The cron job that auto-generates the next day's draft at 8pm Dubai did not fire on 2026-05-25. Last recorded run was May 24 16:00 UTC. The 22 machines in `machines_to_visit` for May 26 were populated during the engine v11 deployment session, not by the cron. Without the cron, no draft is generated for CS to review in the morning.

**Fix:**

1. Investigate why cron job #13 did not fire. Check `cron.job_run_details` for errors or skipped runs.
2. If the RPC it calls (`auto_generate_draft` or the underlying pipeline) errored, fix the error.
3. Add a health check: if `machines_to_visit` for tomorrow is empty after 8:30pm Dubai, send an alert (email or Slack) to CS.

**Acceptance criteria:**

- AC1: Cron job #13 runs successfully every day at 8pm Dubai and generates a draft.
- AC2: If the cron errors, an alert is sent within 30 minutes.

## Bug 5: Engine generates M2W (Remove) without empty-shelf guard

**Severity:** P1 (operational: leaves shelf empty, no substitute)
**Location:** `engine_swap_pod` / `engine_finalize_pod` (backend RPCs)

**Problem:**
The engine generated a "Machine To Warehouse" (M2W) action for MC-2004 B08 (Organic Larder Rice Cake, 24 units) without checking whether a replacement product would fill the shelf. The M2W would have left shelf B08 completely empty after the visit. A shelf should never be left empty after a refill visit unless it is being decommissioned.

This was caught by CS during manual review and cancelled. The engine should have either (a) paired the M2W with a swap-in, or (b) suppressed the M2W and flagged it as a manual decision.

**Fix:**

1. In `engine_finalize_pod`, add a post-processing check: for every M2W/Remove action in the draft, verify that the same shelf has a corresponding Add New or Refill action with qty > 0.
2. If no replacement exists, either:
   - (a) Demote the M2W to a `capacity_mismatch_warning` with reason "no_replacement_for_shelf" and remove it from the executable plan, OR
   - (b) Auto-trigger a swap-in search for the shelf (call the Pearson correlation logic from engine_swap_pod Pass 2).
3. Add a `warnings` output entry: `{ shelf, machine, reason: "m2w_no_replacement", product }` so CS can see it in the advisory.

**Acceptance criteria:**

- AC1: No M2W/Remove action survives finalize without a corresponding replacement action on the same shelf, unless the shelf is tagged for decommission.
- AC2: Orphan M2W rows appear in `capacity_mismatch_warnings` output.
- AC3: FE shows a warning badge on M2W rows that have no paired swap-in.
