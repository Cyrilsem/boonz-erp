Data layer — the four oracles

Four different questions, four different sources. Never confuse them.

Each oracle answers exactly one question about the fleet. The refill engine must query the right one for the right question — mixing them is the #1 source of bad decisions.

📦 Stock oracle — "What's physically on the shelf right now?"
Source: v_live_shelf_stock view, flattening weimi_device_status.door_statuses JSONB
Status: ✅ Live. CC-00 complete. View created Round 1, hotfixed for cabinet selector bug. Verified against NOOK ground truth.

How it works: The refresh-stage1 Edge Function calls Weimi's /ext/device-info endpoint and stores the response as JSONB in weimi_device_status.door_statuses. Structure is array-of-cabinets, each cabinet has layers, each layer has aisles. The view does DISTINCT ON (machine_id) ORDER BY snapshot_at DESC to get only the latest snapshot per machine, then flattens via jsonb_array_elements down to one row per slot.

Columns exposed: machine_id, machine_name, weimi_device_id, cabinet_index, layer_label, aisle_code, slot_name, goods_name_raw, pod_product_id (4-tier match), match_method, current_stock (clamped to ≥0), max_stock, fill_pct, is_broken, is_enabled, price_aed, snapshot_at, is_eligible_machine.

Match resolution: 4-tier pipeline. Direct exact match → case-insensitive match → product_name_conventions alias lookup → unmatched (NULL pod_product_id). Current stats: 87% direct, 6% conventions, 1% case-insensitive, 6% unmatched (only 1 real test product on non-eligible machines).

Refresh cadence: nightly at 23:59 Dubai via cron, plus on-demand via SELECT refresh_fleet_data(90).
⏳ Expiry oracle — "What's about to go bad in the pods?"
Source: pod_inventory, driver-maintained + FIFO sales decrement
Status: ✅ Live. Pre-existing, not modified in Round 1.

How it works: Driver does monthly audit, writes batch + expiration_date rows. refresh-stage1 calls run_pod_inventory_decrement() which subtracts new sales from the oldest-expiring matching batch. So current_stock in this table reflects "what's left in the pod by batch and expiry," not "what's on the shelf."

Consumers: Engine 1 (catalog-level expiry-at-risk list), Engine B (expiry-aware quantity sizing), Engine 2 (rescue move prioritization for batches near expiry).

NOT used for: "how much is in the machine right now" — that's the stock oracle. Confusing these two was the original bug we hit on day 1.
💰 Sales oracle — "What sold, when, and for how much?"
Source: sales_history, live from Weimi via refresh-stage1
Status: ✅ Live. 16,661 rows as of 2026-04-09. Latest transaction within 1 minute of every refresh.

How it works: Weimi's /ext/history-order-page (90 days default lookback) and /ext/today-order-page endpoints. Pulled, deduplicated by tradeNoIn, upserted via upsert_sales_lines RPC.

Consumers: Lifecycle scoring (evaluate-lifecycle nightly), velocity calculation, demand forecasting, correlation matrix.
🎯 Outcome oracle — "Did the refill plan match reality?"
Source: refill_dispatching, populated by field team
Status: ✅ Live. Pre-existing. NEW in v5: explicitly recognized as the learning loop's outcome source.

How it works: When the operator approves a refill plan on /refill, lines flow into refill_dispatching. Driver picks them up via /field/Daily Refill, marks each as packed/picked_up/filled_quantity/returned/return_reason. Engine D's learning loop reads these outcomes via the decision_log.outcome_dispatch_id FK to know whether its quantity sizing, swap candidates, and clubbing decisions actually worked on-site.

Consumers: Engine B (learn from filled_quantity vs sent_quantity gap), Engine C (learn from returned items + return_reason), Engine D (overall learning loop).

Open issues found during implementation

Things we discovered while building Round 1 but haven't fully fixed yet. Each one tracked here so we don't lose them.

A8 over-stocked at NOOK (9/8 = 112%)
After today's refill, NOOK A8 was swapped from "Caprice" to "SF Pancake" but the new product's maxStock wasn't updated to match the actual physical capacity. Result: current_stock exceeds max_stock, fill_pct > 100%. Not a view bug — this is real data from Weimi.

Implication: Engine B (quantity sizing) must handle fill_pct > 100 as a valid state meaning "max_stock is out of date, flag for operator review, don't try to refill." The view returns the over-stocked state honestly; downstream consumers must not crash or wrap around.
Product swap detection at slot level
NOOK A8 went from "Caprice" → "SF Pancake" between snapshots. slot_lifecycle is keyed on (machine_id, shelf_code), not product, so the Caprice lifecycle history silently carries over to SF Pancake. This is contamination analogous to ALHQ → JET-1016 but at the slot level.

Fix needed: evaluate-lifecycle (or a new pre-step) should detect goods_name_raw changes between snapshots and reset the slot's lifecycle score for the new product (with grace period). Add to Phase 1 backlog.
wait_for_request() times out for long-running Edge Functions
refresh-stage1 takes 60-90 seconds. pg_net's default request timeout is shorter (~10s), so net._http_response.status_code stays NULL even though the Edge Function actually completed successfully (visible in get_logs). The function still works (data lands in DB), but our wait_for_request() helper can't programmatically detect completion.

Workaround in use: poll the actual data freshness (e.g., MAX(weimi_device_status.snapshot_at)) instead of net._http_response.

Proper fix later: rewrite wait_for_request() to take a "freshness signal" parameter — a SQL fragment that returns a timestamp — and poll that instead. Low priority. Current workaround is fine.
JET-1016 placeholder fields after Round 2 split
Round 2 CC-09 will split ALHQ → JET-1016 using the repurpose_machine() function. CC doesn't know the real values for JET-1016's location_name, location_type, source_of_supply, so it uses placeholders ("JET location (operator to confirm)", "coworking", "BOONZ").

Action after Round 2 lands: CS to update these fields with real values via the admin UI.
5 other already-repurposed machines never investigated
Earlier query found 5 machines with repurposed_at IS NOT NULL in the DB (besides the ALHQ → JET-1016 case being handled in Round 2). They're all Switched off + repurposed_at set, which is a healthy state, but we never checked whether any of them have lingering data contamination on their old machine_ids. Audit needed after Round 2.
VOXMCC-1009 stale field data preserved by merge
After Round 2's CC-09b merge, the surviving VOXMCC-1009 row (machine_id 148c4fcf...) has two wrong field values: location_type='warehouse' (should be cinema/concession) and pod_location='MINDSHARE-1009-4500-O1' (totally different machine). These were pre-existing bad data on the older row before the merge. The merge logic used COALESCE(new.field, old.field) for these columns, which preserves the old value if it's non-NULL. The values were non-NULL but wrong.

Action: CS to update via admin UI Edit Machine (NOT Repurpose flow). Quick manual fix.

Lesson for future merges: COALESCE preserves the old value if non-NULL, even when wrong. For one-time merges, prefer "new value wins for known-fresh fields" rather than COALESCE.

Lessons learned

Engineering lessons from Phase 0. Each one is a personal-rulebook addition for future prompts.

JWT-in-migration regression (commits 747612c → 220af7e → 0da2861 → cea5aee)
CC, when given an ambiguous prompt about Edge Function auth, "helpfully" hardcoded the anon key into the SQL function body and granted EXECUTE to authenticated. Took 3 follow-up commits to clean up: revert function body to vault read, REVOKE authenticated grants explicitly (because CREATE OR REPLACE FUNCTION preserves ACLs), drop the Authorization: Bearer header (which refresh-stage1 validates as a user token, not a service role token).

Personal rulebook addition: for any prompt that involves credentials, secrets, or auth design, include an explicit "DO NOT alter the following design choices under any circumstances, even if it would simplify the implementation" section listing the load-bearing design decisions.
CREATE OR REPLACE FUNCTION does not reset ACLs
The function body changes, but pre-existing GRANT/REVOKE rules persist. So if a previous migration granted EXECUTE to a role and a new migration replaces the function body without explicitly revoking that grant, the role still has access.

Personal rulebook addition: any migration that uses CREATE OR REPLACE FUNCTION on a security-sensitive function MUST follow it with explicit REVOKE statements for every role that should not have access, plus the GRANTs for the roles that should.
JSONB structure assumptions (door_statuses cabinet selector bug)
I told CC the door_statuses structure was "cabinets → layers → aisles" in shorthand. CC interpreted that as door_statuses.cabinets[] when actually door_statuses IS the cabinets array (no wrapper key). Result: empty view on first build.

Personal rulebook addition: when describing JSONB structure in prompts, always specify the literal top-level type explicitly. Use phrases like "door_statuses is an array of cabinet objects, length N, where each element is...". Never use the conceptual shorthand "X → Y → Z" without specifying the JSON literal at each level.
Verify capabilities before promising to handle a task
I told you "I'll fetch the service role key via my Supabase MCP tools" and then discovered my MCP tools only expose publishable keys (anon + sb_publishable_*), not service_role. By design — which is good security. But I made the promise without checking the tool catalog first, costing us a round-trip.

Personal rulebook addition: before offering to do a step that requires a specific tool capability, verify the tool can actually do it. Especially for security-sensitive operations where the answer might be "no, by design."
Migration filename format matters more than you think (sharpened diagnosis)
Round 2.5 commit c651638 created four migration files. CC's commit summary listed all 8 tasks as complete. None of the SQL was applied to production. Detected during verification: every check returned the pre-Round-2.5 state.

Original (incomplete) diagnosis: "CC used 6-digit integer prefixes instead of timestamps." This was wrong.

Sharper actual diagnosis (operator caught this): CC understood the format requirement and tried to use timestamps. The filenames were like 20260409107500_*.sql where 107500 would parse as 10:75:00 — invalid (minute 75 doesn't exist). Supabase's migration runner rejected them silently as malformed timestamps. The format was right; the time arithmetic was wrong.

Recovery: all four migrations re-applied via Supabase MCP apply_migration from chat. Database state now matches the intended outcome. Files in repo were renamed via git mv to valid timestamps in commit 5872ae2, then re-aligned to match the actual DB-recorded versions (20260410104527/104542/104554/104614) in commit 58efb67.

Personal rulebook addition: every CC prompt creating a migration must include explicit reference to the existing filename format AND a sanity check that the time portion is valid (HH ≤ 23, MM ≤ 59, SS ≤ 59). Sample wording: "Use the same filename version format as existing migrations in supabase/migrations/. Run ls supabase/migrations/ | tail -3 first to verify the format. For Boonz that's YYYYMMDDHHMMSS_snake_case_name.sql, where HHMMSS is a real clock time."

Two-step landmine pattern: the bug had a follow-up landmine — even after the files were renamed to valid timestamps in 5872ae2, the new repo timestamps (20260410090001-090004) didn't match the actual DB-recorded versions (20260410104527/104542/104554/104614) from the chat-applied migrations. supabase db push would have tried to re-apply the repo files as new migrations and failed. Caught during verification, fixed in 58efb67. Personal rulebook addition: after any migration filename rename, query schema_migrations and verify the new filenames match the version recorded there, not just that the new filenames are technically valid.
When CC delivers something that should have been live but isn't, verify the delivery mechanism, not just the deliverable
CC's Round 2.5 PR summary said all 8 tasks complete. The git diff showed migration files committed. The frontend changes were correct. Without verification queries against the database, I would have moved on assuming success.

The bridge between "files exist in the repo" and "changes are live in production" is the migration runner. That bridge can fail silently when the input doesn't match the runner's expectations.

Personal rulebook addition: after every CC migration commit, my verification step is not just "does the SQL look right" but "is the schema actually changed in the live database." For DB schema changes specifically, the verification query is: SELECT version, name FROM supabase_migrations.schema_migrations WHERE version >= '<round-start-timestamp>' ORDER BY version. If the expected migrations don't appear in the table, they weren't applied — regardless of what the commit summary says.
Don't fold unrelated tasks into one CC prompt
Round 2 had two themes: machine identity cleanup (CC-08, CC-09, CC-09b, CC-09c, CC-09d) and venue group taxonomy (CC-01b, CC-02b). I folded both into a single 11-task prompt to avoid a round-trip. CC focused on the primary theme (identity cleanup), executed all 6 of those tasks correctly, and silently dropped both venue_group tasks. CC's PR summary doesn't even acknowledge they existed. No error, no warning — they just didn't happen.

Personal rulebook addition: one theme per CC prompt. If I have a cleanup task and a feature-add task, those are two prompts. The token cost of an extra round-trip is small. The cost of a silently-dropped task is large because you only catch it during verification, by which point the prompt is no longer in the working set and re-running it requires reconstructing context.

Specifically: when I write a CC prompt with a numbered task list, the failure mode is that CC mentally bins them by theme and then drops the smaller bin. So the rule is: if I cannot summarize the prompt's purpose in one sentence, it's two prompts. "Phase 0 Round 2: machine identity, venue groups, guardrails commit" was three things in one sentence — that was the warning sign I missed.
Check for existing functions before creating new ones with the same name
I wrote the CC-08 prompt asking CC to CREATE OR REPLACE FUNCTION public.repurpose_machine(...) with a specific 6-parameter signature. I assumed no function with that name existed. There was an older 8-parameter repurpose_machine from a previous field PWA phase. PostgreSQL's overloading rules meant the new function was created as a separate overload alongside the old one, instead of replacing it. Both now coexist with incompatible semantics (rename-in-place vs atomic-split).

Personal rulebook addition: before writing a CREATE OR REPLACE FUNCTION prompt, query pg_proc to check if anything with that name already exists. If yes, decide explicitly: replace it (drop the old one first), rename it (add a suffix), or use a different name for the new one. Never assume the namespace is empty.

SQL pattern to add to my mental checklist:
SELECT proname, pg_get_function_identity_arguments(oid) FROM pg_proc WHERE proname = '<name>';
Run this before writing any function-creating migration prompt.

Design principles

  1. One direction, top to bottom. No loops within a run. Engine 1 → 2 → Layer 1 → Step 2 → Stage 3 (A, C, B) → Engine D → Engine E.
  2. Four oracles, four questions. Stock = v_live_shelf_stock. Expiry = pod_inventory. Sales = sales_history. Outcomes = refill_dispatching. Never confuse them.
  3. Engine 1 is portfolio-only. Catalog decisions, not slot decisions.
  4. Layer 1 stays pure. Per-machine priority only. Geography enters at Step 2.
  5. Guardrails apply at every engine. Each engine reads only the types relevant to its decision.
  6. Engine D is the only writer. No other engine commits to refill_plan_output.
  7. Shared fit scorer. Engine C and Engine 2 share the same scoring function for consistency.
  8. Stage 0 calculators are reference data, not decisions.
  9. Learning loop is across runs only. Deterministic, debuggable.
  10. Reason codes are mandatory. Every override tagged. Engine D's learning ignores operational reasons (stock_unavailable, logistics_constraint).
  11. Advisor-only forever. Skill never bypasses /refill review.
  12. Downstream pipeline unchanged. Skill writes to refill_plan_output, that's it.
  13. Machine eligibility is a precondition. Decommissioned, repurposed, and non-live machines are filtered out before Layer 1 even runs.
  14. Vault for all secrets. Never embed JWTs in any migration or function body.
  15. Bible updates at every checkpoint. Bible drift is the failure mode we cannot afford.
  16. Human-led operation rate limit. Max 10 slot changes per day fleet-wide, max 2 per machine per cycle, 14 days stability between consecutive changes at the same slot. The engine is a co-pilot, not an autopilot. See portfolio_strategy.md Section 9.
  17. Data facts + operator intent, combined. Data layer dominates unless operator intent field explicitly overrides. Engine 1 reads portfolio_strategy.md as priors and overrides on top of lifecycle scoring.
  18. Interview pattern rule. Strategic interviews (tacit operator knowledge data can't see) use doc-coauthoring 3-stage workflow. Mechanical interviews (rule-based, data-derivable) use draft-first: Claude drafts from data + minimal input, operator corrects. Pattern choice made at interview open based on whether Claude can predict 80% of the answer from existing data.

Phase 2 chapter — deferred enhancements

This section lists features and capabilities explicitly deferred out of Phase 1 scope. They are captured here so future-Claude (and future-operator) know they exist in the roadmap without polluting Phase 1 task lists.

Mode system (Conservative / Balanced / Aggressive)

Phase 1 runs in BALANCED mode only — Engine 1 treats portfolio_strategy.md as priors, overrides on strong data signals. Phase 2 adds:

  • Conservative mode — triggered by war, boycotts, supply chain disruption, force majeure, revenue crisis. Rate limits tighter, phase-outs paused, new trials frozen, extended protection on current placements. Ramp-down period 3-7 days when entering this mode.
  • Aggressive mode — triggered by strong revenue growth, expansion phase, operational slack. Rate limits loosened, more aggressive phase-outs, more experimentation in trial slots. Ramp-up period 3-7 days.
  • Mode transitions are gradual, not binary. When flipping modes, the engine ramps behavior over N days rather than switching instantly. Prevents whipsawing operations team.
  • Mode is operator-set in Phase 2. Future (Phase 3+) may add auto-detection from macro signals: revenue trend over 30 days, news feeds, weather, building closures, vacation calendars, recession indicators.

Top-20 ranking by gross profit

Phase 1 uses top-20 by revenue (price × units) as the stable-stage threshold. Revenue is a proxy. Phase 2 switches to top-20 by gross profit (revenue − product_cost) once cost data is reliable across the full catalog. Requires: complete supplier_product_mapping data, validated cost entries in sales_history.product_cost, and operator confirmation that the cost data is clean.

AI product discovery subsystem

Phase 1 trial entry is manual and reactive — operator discovers new products via personal research, supplier calls, grocery store visits. Phase 2+ vision: AI subsystem scans web, social media, ecom platforms for emerging product ideas and surfaces trial suggestions to the operator with rationale (why this product, which venues, projected archetype). Operator still approves every trial entry. Discovery automated, decision remains human.

Operational quality scoring

Schema change: add boonz_products.operational_quality_score column plus subfields for breakage rate, shelf appeal, load difficulty, driver complaint count. Captures physical/operational attributes that are phase-out drivers independent of sales performance. When the schema change lands, Engine 1 reads the operational quality score as an additional input to phase-out decisions. Can also influence trial criteria (a product that sells well but breaks easily and frustrates drivers may not graduate even with strong data signal).

Mode auto-detection from macro signals

Phase 2 mode is operator-set. Phase 3+ adds optional auto-detection from: rolling 30-day fleet revenue trend, external news feeds (crisis indicators), weather (extreme heat / cold / sandstorm affecting foot traffic), calendar events (Ramadan, Eid, UAE National Day, school holidays), building closure notifications, regional travel/tourism indicators. Each signal contributes a weighted vote for mode classification. Operator can always override.

Full learning loop (Engine D intelligence)

Phase 1 Engine D is a decider + writer, but has no learning loop — it doesn't adjust its own scoring weights based on outcome data. Phase 2 adds the learning loop: reads refill_dispatching.filled_quantity / returned / return_reason and decision_log.operator_action / operator_reason to tune engine weights over time. Excluded reasons from learning: stock_unavailable, logistics_constraint, one_off_business_reason (these are operational, not strategic).

Engine 2 relocation planner (full)

Phase 1 Engine 2 handles simple relocations (move product from machine A to machine B when clearly justified). Phase 2 expands to full cross-machine optimization: multi-hop relocations, load balancing across venue groups, travel-scope-respecting reshuffles, seasonal calendar-aware redistributions.
