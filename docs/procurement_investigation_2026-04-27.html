§1 · Oliver Wyman — Executive Process Assessment

Oliver Wyman — Process Intelligence

We reviewed the full procurement lifecycle from PO creation through inventory receipt. The flow has a solid skeleton — two-leg driver/supplier split, batch expiry tracking, field additions — but carries eight compounding defects that make it brittle in production. Three are operational bugs today. Five are structural risks that will scale badly. We've prioritised them P0 → Feature below.

Current-state flow

§2 · Detailed Bug Findings

P0
B-1 · "Union Coop" (SUP_014) missing from walk-in supplier codes Stax

The FE hardcodes the walk-in set as a TypeScript constant: const WALK_IN_SUPPLIER_CODES = ["SUP_005", "SUP_011"] as const in src/app/(field)/field/orders/new/page.tsx. There are three walk-in suppliers in the database:

SUP_005  Union Coop Dubai   ✓ in list
SUP_011  Carrefour          ✓ in list
SUP_014  Union Coop         ✗ MISSING — treated as email supplier

When a PO is created for "Union Coop" (SUP_014), the system correctly creates a driver_tasks row (since the 2026-04-23 fix), but the confirm dialog shows the 📧 Email badge instead of the 🚗 Driver Task badge, and the system tries to send an email to null (Union Coop has no contact_email). The email call is non-blocking so the PO is saved, but the operator receives a confusing "email sent" status message for a supplier that has no email — and the driver task exists but the operator can't see it was created.

Root cause: walk-in classification lives in FE code, not in the database. Every new walk-in supplier requires a code change and redeployment.

Immediate fix: add SUP_014 to WALK_IN_SUPPLIER_CODES. Structural fix: move classification to suppliers.procurement_type column (see F-1).

P0
B-2 · Multi-batch receiving inserts duplicate PO lines Dara Stax

When the warehouse manager receives a PO with multiple expiry batches per product, the receiving page (src/app/(field)/field/receiving/[poId]/page.tsx) handles them as follows:

Batch 1 → UPDATE purchase_orders SET received_date=..., received_qty=...
             WHERE po_line_id = original_line_id
Batch 2 → INSERT INTO purchase_orders (po_id, supplier_id, boonz_product_id,
             ordered_qty=ORIGINAL, received_qty=batch2_qty, ...)
Batch 3 → INSERT INTO purchase_orders (...)

Each additional batch creates a new row in purchase_orders with the same po_id and boonz_product_id, but a copied ordered_qty from the original line. This means:

  • The orders list groups by po_id and sums ordered_qty — so a 24-unit product with 2 batches shows as 48 ordered
  • line_count inflates — 1 product looks like 2 or 3 lines
  • The tasks page loads PO lines and shows the driver inflated quantities
  • If you receive 3 batches of 5 products, you get 15 rows instead of 5, all with wrong ordered_qty

Root cause: expiry batches should NOT be separate PO lines. They should be separate warehouse_inventory rows (which already exist) linked to the same PO line. The PO line is an order, not a batch record. The correct model is: one PO line per product-per-PO, one warehouse_inventory row per batch.

Fix: remove the INSERT path for batches 2+. A new warehouse_inventory row per batch is already being created — that's the right record. The original PO line should be updated with total received_qty (sum of all batches).

P1
B-3 · warehouse_inventory written directly from FE client Cody Stax

The receiving page does this from a browser client component using the anon/session Supabase client:

const { data: whRows } = await supabase
  .from("warehouse_inventory")
  .insert({
    boonz_product_id: ..., warehouse_stock: ...,
    expiration_date: ..., batch_id: ..., status: "Active", ...
  })

warehouse_inventory is a Constitution-protected entity (Appendix A). Direct writes from the FE client violate Articles 1 and 3. The current permissive RLS policy (warehouse_write_wh_inventory) allows this for all staff including field_staff, which means any field staff user can insert arbitrary inventory rows.

Additionally, purchase_orders is written directly (both UPDATE for original lines and INSERT for extra batches), and inventory_audit_log is written directly for price changes. None of these go through an RPC.

Fix: introduce a receive_purchase_order RPC that accepts the PO ID, an array of line receipts (product, batches, price), and atomically: updates PO lines, creates warehouse_inventory rows, creates audit log entries, and marks po_additions as received. The FE calls this single RPC.

P1
B-4 · po_additions not processed in receiving confirm Stax

The receiving page shows field additions (products added by the driver beyond the original PO) in an amber section, but the handleConfirm() function only loops over lines — never over additions. When the warehouse manager clicks "Confirm receiving", additions remain in po_additions.status = 'pending_receive' indefinitely and no warehouse_inventory row is ever created for them.

The older additions in the database that do show status = 'received' and received_at timestamps were handled via a separate path in the operator console (/app/procurement) — there is a parallel receiving mechanism. This creates two partially overlapping flows:

  • Field receiving page (/field/receiving/[poId]) — handles original PO lines only
  • Operator console — can mark additions as received

Neither flow is complete. The field page should include additions in the confirm batch, with the same batch/expiry/price fields. The RPC proposed in B-3 should handle both.

P2
B-5 · po_number generated client-side (race condition) Dara Stax

The FE generates the next po_number by querying the current max and adding 1:

const { data: lastPo } = await supabase.from("purchase_orders")
  .select("po_number")
  .not("po_number", "is", null)
  .order("po_number", { ascending: false })
  .limit(1).single()
const nextNumber = (lastPo?.po_number ?? 9016) + 1

This runs twice per PO creation (once in fetchData for display, once in handleConfirmSend for actual insertion). If two users create POs simultaneously, or if the user takes a long time between opening the page and confirming, they get duplicate po_numbers. There is no UNIQUE constraint on purchase_orders.po_number.

Fix: use a Postgres sequence or generate the number inside the create_purchase_order RPC using SELECT nextval(...). The display number shown before submission can be approximate / prefixed with "~".

P2
B-6 · No driver → warehouse handoff state Process gap

When a driver marks a task as "collected" in /field/tasks, the PO continues to appear as "Pending" in /field/orders because purchase_orders.received_date is still null. The warehouse manager has no signal that goods are en route or arrived.

The current states in driver_tasks.status are: pending → acknowledged → collected → cancelled. There is no bridge from "collected" to "receiving pending at warehouse". This means:

  • Operator sees PO as open/pending even after driver has collected
  • Warehouse has no way to know what's arriving today
  • If a driver partially purchases, the WH doesn't know the expected shortfall before goods arrive

Fix: add a collected status indicator to the orders view (cross-reference driver_tasks by po_id). When a driver marks a task collected, the orders list should show "In transit — collected by driver" rather than "Pending". The full receive flow completes when the warehouse scans/confirms arrival.

§3 · Dara — Schema Proposals

Dara — Data Architect

D-1 · Add suppliers.procurement_type column (F-1)

Removes the hardcoded WALK_IN_SUPPLIER_CODES array from the FE. The column drives both the confirm dialog UX and the notification routing.

-- Migration: procurement_supplier_type
ALTER TABLE public.suppliers
  ADD COLUMN IF NOT EXISTS procurement_type text NOT NULL DEFAULT 'supplier_delivered'
  CHECK (procurement_type IN ('supplier_delivered', 'walk_in'));

COMMENT ON COLUMN public.suppliers.procurement_type IS
  'walk_in: driver purchases on behalf (Union Coop, Carrefour).
   supplier_delivered: supplier delivers to warehouse (email PO sent).';

-- Backfill from known codes
UPDATE public.suppliers
SET procurement_type = 'walk_in'
WHERE supplier_code IN ('SUP_005', 'SUP_011', 'SUP_014');

CREATE INDEX idx_suppliers_procurement_type
  ON public.suppliers(procurement_type)
  WHERE procurement_type = 'walk_in';

The FE query then becomes: select('supplier_id, supplier_name, supplier_code, contact_email, procurement_type') and isWalkIn = supplier.procurement_type === 'walk_in'. No more hardcoded codes.

D-2 · Add po_number sequence (B-5)

-- Migration: procurement_po_number_sequence
CREATE SEQUENCE IF NOT EXISTS public.po_number_seq
  START WITH 9100
  INCREMENT BY 1
  NO MAXVALUE
  NO CYCLE;

-- Backfill to current max
SELECT setval('public.po_number_seq',
  GREATEST(COALESCE((SELECT MAX(po_number) FROM public.purchase_orders), 9099), 9099)
);

The create_purchase_order RPC will call nextval('public.po_number_seq') to assign the number atomically. No client-side max query needed.

D-3 · Schema note on purchase_orders multi-batch model

The correct model (to fix B-2) is:

  • purchase_orders — one row per product per PO. ordered_qty = original order. received_qty = total received (sum of all batches). received_date = date of first batch receipt.
  • warehouse_inventory — one row per expiry batch. Already exists and is already inserted correctly. The batch reference back to the PO is via batch_id which encodes the po_id.

No schema change is needed for the PO line model — the bug is in the FE logic that inserts extra rows. The receiving RPC simply does not insert extra PO lines; it only inserts additional warehouse_inventory rows.

  • D-2: Sequence is outside any protected entity. ✅ ready for Cody fast-path.

§4 · Stax — FE / Wiring Plan

Stax — Full-Stack Engineer

S-1 · Immediate patch: B-1 walk-in code fix

File: src/app/(field)/field/orders/new/page.tsx

Before the suppliers query loads, the page already fetches procurement_type (once D-1 migration is applied). Replace the hardcoded constant:

// BEFORE (remove this)
const WALK_IN_SUPPLIER_CODES = ["SUP_005", "SUP_011"] as const;
const isWalkIn = WALK_IN_SUPPLIER_CODES.includes(supplier.supplier_code);

// AFTER (use DB field)

The supplier SELECT query must include procurement_type in its column list. This is a read-only change — no protected entity write, no RPC needed for this field. Rule S2.

S-2 · New RPC: create_purchase_order

Replaces: FE direct insert to purchase_orders + FE direct insert to driver_tasks + FE insert to po_notifications. Atomically:

  • Gets next po_number from sequence
  • Inserts all PO lines into purchase_orders
  • Inserts one row into driver_tasks
  • Inserts one row into po_notifications
  • Returns { po_id, po_number }

FE call after migration:

const { data, error } = await supabase.rpc('create_purchase_order', {
  p_po_id: poId,
  p_supplier_id: supplierId,
  p_purchase_date: poDate,
  p_created_by: user.id,
  p_lines: finalLines.map(l => ({
    boonz_product_id: l.product_id,
    ordered_qty: l.qty,
    price_per_unit_aed: l.price ?? null,
    expiry_date: l.expiry_date || null,
  })),
});

Email notification stays as a non-blocking Edge Function call after the RPC returns. The RPC only handles the DB writes. Rules S1, S2, S7.

S-3 · New RPC: receive_purchase_order

Replaces: all direct writes in /field/receiving/[poId]/page.tsx. Accepts the po_id and an array of line receipts with sub-batch arrays. Atomically:

  • Updates each purchase_orders row: sets received_qty (sum of all batches), received_date, price_per_unit_aed
  • Inserts one warehouse_inventory row per expiry batch (correct — already happening, just moved to RPC)
  • Does NOT insert extra PO lines for extra batches — this fixes B-2
  • Updates po_additions with status 'received', received_at, received_by for all pending additions
  • Inserts warehouse_inventory row for each addition (fixes B-4)
  • If price changed, writes to inventory_audit_log

FE call:

const { error } = await supabase.rpc('receive_purchase_order', {
  p_po_id: poId,
  p_received_by: authUser.id,
  p_lines: lines
    .filter(l => !l.received_date)
    .map(l => ({
      po_line_id: l.po_line_id,
      price_per_unit_aed: effectivePrice(l),
      wh_location: l.wh_location || null,
      batches: l.batches
        .filter(b => b.received_qty > 0)
        .map(b => ({
          received_qty: b.received_qty,
          expiry_date: b.expiry_date || null,
        })),
    })),
  p_additions: additions
    .filter(a => a.status === 'pending_receive')
    .map(a => ({
      addition_id: a.addition_id,
      boonz_product_id: a.boonz_product_id,
      qty: a.qty,
      price_per_unit_aed: a.price_per_unit_aed ?? null,
      wh_location: null,
    })),
});

S-4 · Orders list: show driver task status (B-6)

Files: /field/orders/page.tsx (driver view) and /app/procurement/page.tsx (operator view). Both should join driver_tasks by po_id to show the current collection state alongside the receiving state. Add to the existing fetch:

// Fetch driver task status per PO
const { data: taskData } = await supabase
  .from("driver_tasks")
  .select("po_id, status, collected_at, outcome")
  .in("po_id", poIds);

const taskMap = new Map(taskData?.map(t => [t.po_id, t]) ?? []);

Then in the UI, a PO card shows:

  • 🟡 Pending → driver hasn't acknowledged yet
  • 🔵 On my way → driver acknowledged
  • 🟢 Collected (awaiting WH receive) → driver marked collected, WH not received yet
  • ✅ Received → WH confirmed receipt (received_date set)

Cody handoff from Stax

Two new canonical writer RPCs for Cody to review:

  • create_purchase_order — writes to purchase_orders, driver_tasks, po_notifications. Must: set app.via_rpc + app.rpc_name, validate inputs, validate caller role (field_staff|warehouse|operator_admin|superadmin|manager).
  • receive_purchase_order — writes to purchase_orders, warehouse_inventory, po_additions, inventory_audit_log. Must: same Article 4 requirements. This is the single canonical write path for goods receipt.

§5 · Cody — Constitutional Review

Cody — CTO Advisor / Constitutional Review
Verdict on current procurement flow
⚠️ Multiple violations active in production. The receiving page writes directly to warehouse_inventory (protected entity, Appendix A) from an authenticated browser client. This is a clear Article 3 violation. Two new canonical writers are needed and must pass the full Article 4 checklist before shipping. purchase_orders and driver_tasks should be nominated for Appendix A as part of this migration — they are procurement-critical and should have the same write-path guarantees as other protected entities.

Articles checked & findings

BLOCK
Article 3 — warehouse_inventory direct write from FE client

warehouse_inventory is in Constitution Appendix A. The current permissive RLS policy (warehouse_write_wh_inventory) allows writes from all staff roles including field_staff. This policy must be tightened once the RPC is in place. The receiving page must be refactored to call receive_purchase_order (Stax S-3) before this path is acceptable.

FLAG
Article 1 — purchase_orders has no canonical write path

purchase_orders is written from three places: the new-order page, the receiving page (update + insert), and potentially the operator console. There is no canonical writer RPC. Once create_purchase_order and receive_purchase_order are in place and purchase_orders is added to Appendix A, the direct writes become Phase B cleanup targets.

FLAG
Article 5 — driver_tasks.status has no state machine enforcement

The tasks page does supabase.from("driver_tasks").update({status: "acknowledged", ...}) directly from the browser. Status transitions are not validated — any client could set status to any value. Once driver_tasks is added to Appendix A, status transitions must go through the canonical writer. For now, this is flagged but not a hard block given the small user population.

APPROVE
Articles 2, 7, 12, 14 — Schema proposals (Dara D-1, D-2)

D-1 (suppliers.procurement_type): forward-only DDL, suppliers is not in Appendix A, RLS already enabled, no audit log concern. ✅ Approve. Migration name: procurement_supplier_type_column.

D-2 (po_number_seq): sequence creation is non-protected DDL. ✅ Approve. Migration name: procurement_po_number_sequence.

Next actions for the two new RPCs

Before create_purchase_order and receive_purchase_order can ship, each must satisfy the full Article 4 checklist:

  • PERFORM set_config('app.via_rpc', 'true', true); at the top of BEGIN
  • PERFORM set_config('app.rpc_name', '<function_name>', true);
  • NULL checks on all required parameters (p_po_id, p_supplier_id, p_lines, etc.)
  • Role validation: EXISTS (SELECT 1 FROM user_profiles WHERE id = auth.uid() AND role = ANY(ARRAY['field_staff','warehouse','operator_admin','superadmin','manager']))
  • The generic audit_log_write trigger must be installed on purchase_orders and warehouse_inventory (already installed on warehouse_inventory; verify for purchase_orders)
  • Add entries to RPC_REGISTRY.md and CHANGELOG.md
  • Propose Constitutional Amendment to add purchase_orders and driver_tasks to Appendix A (Article 15)
