Executive summary

What this is: a refill engine that lives as a Claude skill, plugs into the existing Supabase pipeline, and writes plans into refill_plan_output. Operator review, dispatching, and field execution stay unchanged downstream.

How it thinks: portfolio decisions first (catalog-wide), then per-machine planning (locked set decided by priority + geography), then assembly by Engine D, then routing by Engine E.

What's new compared to today: learning loop across runs, codified guardrails by type, lifecycle-aware decisions, fleet-wide relocation planning, expiry-driven rescue moves, vitrine quality cap.

What we're not changing: the /refill review UI, the refill_dispatching table, the /field/Daily Refill module, the field team workflow.

Data layer — the three oracles

Three different questions, three different sources

Each of these answers one question about the fleet. Using the wrong source for the wrong question is the #1 source of bad decisions — the refill engine must never conflate them.

📦 Stock oracle — "What's physically on the shelf right now?"
Source: Weimi live, via weimi_device_status.door_statuses JSONB
Populated by refresh-stage1 Edge Function which calls Weimi's /ext/device-info endpoint. Returns full cabinet → layer → aisle structure with currStock, maxStock, isBroken, isEnable, goodsName, price, weight per aisle. This is the live stock source.

Access pattern: flatten the JSONB through a view (v_live_shelf_stock, to be built in CC-00) into per-slot rows. Match goodsName to pod_products.pod_product_name via product_name_conventions normalization.

Freshness: as fresh as the last refresh-stage1 run. Automated via CC-00c cron + on-demand via CC-00b from Claude.
⏳ Expiry oracle — "What's about to go bad in the pods?"
Source: pod_inventory, driver-maintained + FIFO sales decrement
Tracked manually by the driver during monthly audits. Every row has expiration_date and batch_id. The run_pod_inventory_decrement() function (part of refresh-stage1) subtracts each sale from the oldest-expiring matching batch so the remaining stock still reflects what's actually in the pod.

Consumers: Engine 1 (expiry-at-risk list), Engine B (expiry-aware quantity sizing), Engine 2 (rescue move prioritization).

NOT used for: "how much is in the machine right now" — that's the stock oracle above. pod_inventory is about WHICH batches are in the pod and WHEN they expire.
💰 Sales oracle — "What sold, when, and for how much?"
Source: sales_history, live from Weimi via refresh-stage1
Weimi's /ext/history-order-page and /ext/today-order-page endpoints. Sync pulls 90 days of history + today's orders on every refresh. Trustworthy, up to the minute.

Consumers: lifecycle scoring, velocity calculation, demand forecasting, correlation matrix.
⚠️ Legacy: v_machine_shelf_plan + refill_instructions
To be deprecated in CC-00
Current implementation of v_machine_shelf_plan reads current_stock from refill_instructions, which is a legacy AI-generated table written only twice (Jan 19, Mar 31). Stale by design. After CC-00 lands, v_machine_shelf_plan will be rewritten to flatten weimi_device_status.door_statuses JSONB, OR a new v_live_shelf_stock view will replace it entirely.
❌ Dead: weimi_aisle_snapshots + v_current_aisle_inventory
Schema exists, nothing writes to it — orphaned effort
These were an intended normalized/flattened alternative to the JSONB in weimi_device_status.door_statuses. Never wired up. Can be dropped or left dormant. Do NOT reference from any new code.

Guardrail library

⚡ Guardrails by type — each engine reads only the types it needs

A single library of markdown files under /guardrails/*.md, organized by rule type (not by engine). Each engine in the pipeline declares which types it consults.

Read by: Engine 1

  • Double-down mandates
  • New SKU grace periods
  • Brand strategic locks
🚚 Travel scope rules
Read by: Engine 2, Engine C
  • VOX products → only VOX machines
  • BOONZ products → only non-VOX machines
  • LLFP products → specific machines only
  • Fixed-machine products
  • Category restrictions
🚫 Coexistence rules
Read by: Engine 2, A, C, D
  • Coca + Pepsi forbidden at Addmind, Ush
  • Same product_family auto-block
  • Brand exclusivity contracts per cluster
  • Forbidden pairs (global or per-machine)
🧱 Layout rules
Read by: Engine A
  • 70/30 stability rule
  • VOX hard-locks (V0 machines = no rearranging)
  • Max swap rate per machine per week
  • Promote/demote constraints
  • Anchor slot rules
📦 Quantity rules
Read by: Engine B
  • Min facings per signal tier
  • Vitrine minimums (visual floor)
  • Max stock per slot
  • Expiry-aware sizing
  • Reserved stock priority (FIFO)
🗺️ Routing rules
Read by: Engine E, Step 2
  • Time budget defaults & overrides
  • Opening hours per machine
  • Parking restrictions, delivery windows
  • Building groupings (clubbing)
  • Zone definitions
⚖️ Decision rules
Read by: Engine D
  • Objective: waste-first, then revenue
  • Conflict resolution priority order
  • Override hierarchy
  • Bonded pair rules (same-day vs split-day)
  • Hard safety guarantees
⏰ Vitrine cap rules
Read by: Layer 1
  • Default 10-day vitrine cap
  • Per-machine overrides in vitrine_machines.md
  • Slow-machine extensions
  • Bad-logistics exceptions
🎯 Machine eligibility rules
Read by: Layer 1 (precondition)
  • Include only: adyen_status = 'Online today' AND adyen_inventory_in_store = 'Live'
  • Exclude: repurposed_at IS NOT NULL
  • Exclude: WH3_* pending-setup warehouse machines
  • Exclude: non-functional machines

Learning loop across runs

🔁 Memory-based learning, not ML

Engine D writes a decision_log row for every plan line: which engine made which call, what inputs drove it, what the operator did (accept / override / reject), the override reason code, and the eventual outcome from refill_dispatching (filled_quantity, returned, return_reason).

Reason codes (locked at design time):

• disagree_with_logic — operator thinks engine is wrong (strong learning signal)
• stock_unavailable — operational, no learning weight
• logistics_constraint — operational, no learning weight
• test_override — manual experiment, no learning weight
• one_off_business_reason — exceptional, light learning weight
• other — flagged for review, no automatic learning

On the next run, every engine reads the log. disagree_with_logic overrides accumulate as signals. After N occurrences (default 3) the engine adjusts its ranking. Operational overrides are ignored for learning. This prevents the engine from getting dumber every time you make an exception.

Outcomes flow automatically: refill_dispatching.filled_quantity tells Engine B if its qty sizing was right. returned + return_reason tells Engine C if its swap candidates actually worked on-site.

Known limitations & mitigations

⚠️ Accepted with mitigations

  1. Industry-specific seasonality data doesn't exist yet. Build as add-on later, engine works without it.
  2. Machine similarity is statistically thin at 32 machines. Engine C uses similarity as tiebreaker only.
  3. Product correlation needs volume. Filter aggressively to high-confidence pairs only.
  4. Learning loop can over-fit on operator overrides. Reason codes + 3-occurrence threshold prevent this.
  5. Stage 0 calculators are expensive. Weekly cron, not per-run.
  6. 10-day vitrine cap is hard-coded. vitrine_machines.md overrides from day 1.
  7. Engine 2 bonding logic is tricky. Write tests, assertion checks in Engine D.
  8. Strategic intent isn't in any database. portfolio_strategy.md captured in CS-01 before Engine 1 builds.
  9. Goods name matching between Weimi and pod_products. Uses product_name_conventions normalization (same logic as evaluate-lifecycle). Watch for unmapped SKUs during v_live_shelf_stock rollout.

Design principles

The non-negotiables

  1. One direction, top to bottom. No loops within a run.
  2. Three oracles, three questions. Stock = weimi_device_status JSONB. Expiry = pod_inventory. Sales = sales_history. Never confuse them.
  3. Engine 1 is portfolio-only. Catalog decisions, not slot decisions.
  4. Layer 1 stays pure. Per-machine priority only. Geography at Step 2.
  5. Guardrails apply at every engine. Not just D. Each engine reads only the types relevant to it.
  6. Engine D is the only writer. No other engine commits to the database.
  7. Shared fit scorer. Engine C and Engine 2 share consistency.
  8. Stage 0 calculators are reference data, not decisions.
  9. Learning loop is across runs only. Deterministic, debuggable.
  10. Reason codes are mandatory. Every override tagged.
  11. Advisor-only forever. Skill never bypasses /refill review.
  12. Downstream pipeline unchanged. Skill writes to refill_plan_output, that's it.
  13. Machine eligibility is a precondition. Decommissioned, repurposed, and non-live machines are filtered out before Layer 1 even runs.
